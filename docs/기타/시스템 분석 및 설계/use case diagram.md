# use case diagram

## use case

 * `사용자의 시점`에서 시스템을 모델링하는 역할이다.
 * `요구 사항`을 알아내고 파악하는 과정이다.
 * 시스템 분석가가 사용자와 힘을 합쳐 시스템의 사용 방법을 결정하는데 도움을 주는 도구이다.
 * 시스템 사용자에 대한 시나리오의 집합이다. 사용자가 시스템을 사용하는 절차를 정리한다.

## use case의 목적

::: tip
유스 케이스는 시스템을 사용하는 사용자(실제 시스템을 사용하는 사용자)에게 정보를 얻어내는데 매우 유용하다. 
:::

 * 시스템 사용자를 `시스템 분석과 설계`의 `초기 단계`에 포함시킨다. 사용자들이 진정으로  도움을 받을 수 있는 시스템을 만들 확률이 높아진다. (서비스 지향적인 접근 방식으로 문제를 해결한다.)

### Actor

 * 시스템과 데이터 혹은 명령을 주고 받는 것 중 시스템 내부의 것이 아닌 것이다. 즉 `시스템의 외부에서 명령을 주고 받는다.`
 * 시스템의 `사용자`이다. 또한 소프트웨어나 하드웨어가 될 수 있다.

<p align=center>
    <img src=https://user-images.githubusercontent.com/59357153/141668907-2efe7fd2-a84b-4d97-94f1-bd2203e8e64e.png>
</p>

고객은 직원과 대화를 통해 예약을 진행한다고 가정한다. 여기서 예약 시스템을 사용하는 것은 `직원`이다.
 * 고객은 예약 시스템의 `Actor가 아니다.`
 * 직원은 예약 시스템의 `Actor가 된다.`

고객도 시스템을 직접 사용하면 `Actor가 될 수 있다.`

<p align=center>
    <img src=https://user-images.githubusercontent.com/59357153/141668922-24d16414-ca42-4fc5-8323-b001d0c2dd6f.png>
</p>

이러한 엑터는 2가지 종류를 가진다.

1. `primary actor`: 시스템을 사용하고 직접적으로 이득을 얻는 엑터이며 시스템의 왼쪽에 그려진다.
2. `secondary actor`: primary actor가 이득을 얻기 위해 도움을 주는 엑터로 보통 외부 시스템을 의미하며 `<<actor>>`를 표기하여 엑터임을 나타낸다. 보통 시스템의 오른쪽에 그려준다.

<p align=center>
    <img src=https://user-images.githubusercontent.com/59357153/141670089-2a2743ec-64e9-4448-ad21-da0f8fc5f42d.png>
</p>

::: tip
엑터 이름은 구체적이면 안된다. 엑터 이름을 구체적으로 설정하게 되면 해당 엑터만 시스템을 사용한다는 의미가 될 수 있기 때문에 보다 `추상적인 이름`을 사용한다. 

ex) 만약 엑터를 홍길동으로 설정할 경우 홍길동만 사용할 수 있는 시스템으로 한정하여 해석될 수 있다.
:::

### use case

엑터가 시스템을 사용하는 하나의 시나리오이다. 유스케이스의 자세한 시나리오 절차는 다이어그램에 포함되지 않는다. 단순히 `이름`만 표기한다. 어떻게 구현할 것인지에 대하여 신경쓰지 않아도 된다. 최대한 `단순하게 표현`한다.

<p align=center>
    <img src=https://user-images.githubusercontent.com/59357153/141668873-c0037633-1331-4351-aacb-4a44e73d57e6.png>
</p>

`유스 케이스`는 `엑터가 관심`을 가지고 있는 `유용한 일을 달성하기 위한 시나리오의 집합`을 명시한다. 예를 들면 예약 시스템에는 예약을 진행하고 예약을 조회하는 등 다양한 기능이 제공되어야 한다.

### relationship

#### 1. 포함(inclusion) 관계

다른 유스 케이스에서 기존의 유스 케이스를 재사용하고 있는 관계이다.

특정 상품을 예약하기 위해서는 사용자 인증이 필요하다고 가정한다. 사용자 인증을 위한 기존 유스 케이스를 포함하는 관계를 표시할 수 있다. 사용자 인증은 상품 예약을 위해서는 `반드시` 실행해야 한다.

<p align=center>
    <img src=https://user-images.githubusercontent.com/59357153/141670301-8bf9bbc0-9e7a-48f3-991e-8d814288fd55.png>
</p>

#### 2. 확장(extension) 관계

기존의 유스 케이스에 진행 단계를 추가하여 새로운 유스 케이스로 만들어낸 관계이다.

이미 존재하는 시스템에 `별도의 기능을 추가`하는 관계를 나타낸다. 여기서 `extend`한 유스케이스는 `선택사항`이다. 아래 예시를 보면 예약 리스트를 조회할 때 브랜드 별 검색`할 수도 안할 수도` 있다.

<p align=center>
    <img src=https://user-images.githubusercontent.com/59357153/141670996-4d8c3aa4-f8be-48f8-9dd1-8cf7804a2c42.png>
</p>

#### 3. 일반화(generalization) 관계

한 유스 케이스가 다른 유스 케이스를 `상속한 관계`이다. 보편적인 유스케이스와 구체적인 유스케이스 사이에서 존재한다. `엑터`도 `일반화`가 가능하다. 

아래는 `예약 상품 검색`을 `확장`한 개념인 `상품 이름 검색`과 `상품 고유 번호 검색`이 존재한다. 실제 행위를 위해서는 예약 상품 검색을 상속한 `상품 이름 검색`과 `상품 고유 번호 검색` 중 `반드시 선택`하여 진행해야 한다.
<p align=center>
    <img src=https://user-images.githubusercontent.com/59357153/141671213-9fc359cd-4570-41d7-9c26-d1e94b5ce02f.png>
</p>

::: tip 
`확장 관계`와 `일반화 관계`는 유사한 부분이 많다. 상황에 따라 모두 활용이 가능하기 때문에 정답은 존재하지 않는다. 

보통 `확장 관계`는 별도의 기능인 유스 케이스를 `option`으로 사용한다. 즉 사용하지 않는 경우를 포함한다. 만약 적어도 하나의 `option`이 필요한 경우에는 `일반화 관계`를 사용하여 표현한다.
:::

#### 4. 그룹화(grouping) 관계

여러 개의 유스 케이스를 조직화하는 단순한 방법이다. 유스 케이스 다이어그램이 여러 개의 유스 케이스를 가지고 있는 형태일 때 유스 케이스들을 조직화한다. 

관련된 유스 케이스를 하나의 패키지로 그룹화를 진행할 수 있다.

## 시스템 분석 단계에서 use case diagram

 * 의뢰인과의 인터뷰로 시작한다. 인터뷰를 통해 시스템 분야에 대한 지식의 기초가 되는 유스 케이스 다이어그램을 그린다. 
 * 사용자와 인터뷰를 진행한다. 엑터와 추상적 수준의 유스 케이스 다이어그램을 그려냄으로써 기능적인 요구사항을 정리할 수 있다. 또한 지속적인 사용자와의 인터뷰를 통해 시스템 요구사항을 구체화한다.

## use case diagram 팁

 1. 단순하게 유지해라. 유스 케이스는 자주 변경되는 사용자 인터페이스 요소는 배제하고, 사용자 관점에서 시스템의 행위에 초점을 맞춰야 한다. 그렇기 때문에 단순하게 유지하는 것이 좋다.
 2. 너무 적은 것이 너무 많은 것보다 낫다. 부족하면 나중에 더 추가할 수 있다.
 3. 한 가지만 기억해라. "내일이면 다 바뀐다. [로버트 마틴]"
